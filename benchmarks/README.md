# mcp-contract benchmarks

Four runnable benchmarks against public MCP-attack scenarios. Phases 1-2
measure **detection** (recall); Phase 3 measures a **static** poisoning signal;
Phase 4 measures **specificity** (false-positive rate on benign traffic) — a
detector is only useful if it also stays quiet.

- **Phase 1** — a labelled, reproducible detection matrix through the shipped
  `mcp-contract verify` gate. **No Docker, no LLM, no network.**
- **Phase 2** — real Docker containers through `mcp-contract run --backend
  docker --egress-proxy`, asserting on the events the adapter actually
  observed/enforced. **Docker required, no LLM.**
- **Phase 3** — a static PIE coverage baseline on real MCP servers plus a
  tool-poisoning tripwire measured on the real MCPTox corpus. **Offline.**
- **Phase 4** — benign multi-step / multi-server workflows on real servers;
  measures the false-positive rate (must be 0 hard FPs). **Offline.**

```bash
pip install -e ".[dev]"

# Phase 1: deterministic detection matrix (byte-stable, CI-able)
python benchmarks/run.py          # prints the matrix; exit 0 iff all pass
python benchmarks/run.py --json   # also writes benchmarks/results.json

# Phase 2: real containers + real network enforcement
MCP_CONTRACT_DOCKER_TESTS=1 python benchmarks/run_docker.py

# Phase 3: PIE coverage + tool-poisoning tripwire
bash benchmarks/fetch_mcptox.sh   # optional: pull the real 485-case MCPTox corpus
python benchmarks/run_pie.py      # falls back to a synthetic sample if not fetched

# Phase 4: benign-workflow false-positive / specificity matrix
python benchmarks/run_benign.py
```

## Phase 1 — deterministic detection matrix

Runs the **shipped `mcp-contract verify` path** over hand-authored
`(manifest, approved-policy, event-trace)` triplets and scores the real exit
code against an expected verdict. It replays event traces through the
mock/verify path, so it is byte-stable and CI-able.

## Why a *partitioned* matrix and not one "detection rate"

`mcp-contract` is a **runtime capability-drift monitor**, not a prompt-injection
detector. Public MCP-attack benchmarks (MCPTox, MCPSecBench, MCP-SafetyBench)
mostly measure an **LLM agent's** attack-success rate — a different threat
model. Collapsing them into a single "we caught X%" number would be dishonest,
so every case is tagged with the partition it belongs to and scored against what
the tool is *actually designed* to do:

| Partition | What it is | Correct behavior | Gate |
|---|---|---|---|
| `in-scope` | server exercises a capability **class it never declared** (arith server opens a socket, net-only server reads `~/.ssh/id_rsa`) | **hard-fail** | `verify` exit **1** (`outside_contract`) |
| `manifest-tamper` | manifest changed after the policy was approved (rug-pull) | **hard-fail** | `verify` exit **2** (hash mismatch) |
| `soft` | declared class, **scope** violated (fs server reads `/etc/passwd`; fetch server dials the wrong host) | **soft-flag, gate passes** | exit **0** + `within_manifest_not_policy` |
| `out-of-scope` | agent is manipulated but the resulting tool call **stays within contract** | **stay silent** (correct) | exit **0** |
| `benign` | normal workflow | **stay silent** (false-positive check) | exit **0** |

The honest headline is four numbers, not one: **in-scope recall**, **rug-pull
recall**, **false-positive rate on benign**, and the count of **out-of-scope**
attacks correctly left un-flagged.

### The `soft` partition is the important honesty

`verify` only hard-fails on `outside_contract` — a capability **class** the
manifest never declared. A **scope** violation *within* a declared class
(path traversal on a filesystem server, egress to the wrong host on a fetch
server) is surfaced as `within_manifest_not_policy` and the gate still exits 0.
That is by design in v0:

- **filesystem scope** is enforced at boot by the Docker adapter's ro/rw bind
  mounts, and there are no per-open fs events on that backend, so `verify`
  cannot gate on it — it can only soft-flag a replayed trace.
- **network scope** is hard-enforced by the **egress proxy** (`--egress-proxy`,
  deny-by-default, 403 on miss), not by the `verify` gate.

So `path-traversal-fs` and `mitm-fetch` show up as `PASS (+1 soft-flag)`: the
matrix asserts the *documented* behavior (exit 0 + a soft flag), and points at
Phase 2 for where they become hard blocks.

## Layout

```
benchmarks/
  cases.yaml        # Phase 1 registry: metadata + approve block + expected exit
  manifests/        # declared (benign) tools/list per server type
  events/           # one runtime trace (JSONL) per Phase 1 case
  policies/         # approved policies, regenerated by run.py (git-ignored)
  run.py            # Phase 1: infer -> approve -> verify -> score
  run_docker.py     # Phase 2: build image -> run --backend docker -> assert
  docker/
    Dockerfile.probe   # tiny alpine+curl "server" image
    scenarios/*.sh     # in-container behaviors (egress attempts)
  run_pie.py        # Phase 3: PIE coverage + poisoning tripwire
  poison_samples.jsonl  # committed synthetic poisoned tools (offline fallback)
  fetch_mcptox.sh   # pull the real MCPTox corpus into data/ (git-ignored)
  data/             # downloaded corpora (git-ignored, not vendored)
  run_benign.py     # Phase 4: benign-workflow FP matrix + fleet verify
  benign/
    benign_cases.yaml  # benign workflow registry (manifest + approve + expect)
    manifests/*.json   # real public servers + a faithful yfinance manifest
    events/*.jsonl     # benign multi-step traces
    fleet.yaml         # multi-server benign fleet (fleet verify -> exit 0)
    policies/          # regenerated by run_benign.py (git-ignored)
```

Each case in `cases.yaml` names a `manifest`, an `approve:` block (which caps a
human grants and at what scope), an `events` trace, and an `expect_exit`.
`run.py` infers the policy from the manifest, applies the approval (so the
manifest hash matches — except deliberately for `rug-pull`, which verifies
against `verify_manifest: filesystem-tampered.json`), and runs `verify`.

| `rug-pull` | MCPSecBench — Rug Pull |

## Phase 2 — real containers, real enforcement (Docker)

`run_docker.py` boots real Docker containers through the shipped
`mcp-contract run --backend docker [--egress-proxy]` path and asserts on the
events the adapter actually observed/enforced. It targets the docker backend's
genuine strength — **network enforcement** — and is gated on Docker being
present plus `MCP_CONTRACT_DOCKER_TESTS=1` (the same gate the docker
integration tests use).

```bash
MCP_CONTRACT_DOCKER_TESTS=1 python benchmarks/run_docker.py
```

| Scenario | What it boots | Enforcement proven |
|---|---|---|
| `egress-allowlist` | fetch server, `net.http:[example.com]`, `--egress-proxy`; container curls one allowed + one blocked host | allowed host **tunnels** (`allowed=true` → `within_policy`); blocked host is **403'd by the proxy before any socket opens** (`allowed=false` → `within_manifest_not_policy`) |
| `boot-network-none` | arith server (no `net.http`); container attempts egress | booted with `--network none`; egress **refused by the kernel** — the poller observes no outbound connection at all |

Two honest v0 findings this harness surfaces and works around:

1. **`proc.spawn` observation is coarse.** The `docker top` poller flags the
   container's *own* processes (`sh`, `curl`, …) as `proc.exec` drift unless the
   policy grants `proc.exec`. A real server runs its own process, so these
   scenarios grant `proc.exec` and assert **only** on `net.connect`.
2. **Docker sees IPs, not hostnames.** The same allowed connection is reported
   twice — hostname-level by the proxy (`example.com` → `within_policy`) and
   raw-IP by the `/proc/net/tcp` poller (the container→proxy hop to the docker
   gateway IP → `within_manifest_not_policy`, since an IP matches no hostname in
   the allowlist). This is the documented dual-observation, not a bug.

`fs.open` is deliberately **not** asserted here: the docker backend enforces
filesystem scope at boot via ro/rw bind mounts and emits no per-open events, so
the Phase 1 fs cases (path traversal, SSH-key read) have no runtime signal on
this backend — they stay in Phase 1.

## Phase 3 — PIE coverage + tool-poisoning tripwire (static, offline)

`run_pie.py` has two parts, both deterministic and offline.

**Part A — PIE coverage on real servers.** Runs the shipped policy inference
over the real public manifests vendored in `demo/artifacts/pub_*.json`
(filesystem, fetch, memory, time, sequential-thinking, everything — 40 real
tools). Every capability PIE infers on these real servers lands as
`needs_review`, not `inferred`: the human-approval gate is on by default, so
nothing is auto-granted (0 granted / 7 need review across 6 servers).

**Part B — tool-poisoning tripwire.** For each MCPTox poisoned tool, infer a
policy from the bare tool **name** vs. the name **plus the poisoned
description**; any capability class the description *adds* is one the poisoning
injected into an innocuous-looking tool. Measured on the **real MCPTox corpus**
(485 poisoned tools, 45 servers; [arXiv:2508.14925](https://arxiv.org/abs/2508.14925)):

| Metric | Value |
|---|---|
| tripwire fires (description injects ≥1 capability class) | **201 / 485 = 41.4%** |
| injected capability breakdown | `net.http`×178, `proc.exec`×34, `env`×18 |
| by attack paradigm | Template-1 ×22, Template-2 ×46, Template-3 ×133 |

This is a **static** tripwire — it flags a poisoned description before the
server ever runs, complementing the runtime monitor. It is honest about its
reach: a payload that abuses **already-declared** tools (e.g. `write_file` to
`~/.ssh/id_rsa` on a filesystem server) injects no new capability *class*, so it
does not fire here — that residue is exactly the runtime arm's job (BCM scope
check, the Phase 1 `soft` partition) and motivates the optional `LLMAssist`
path for deeper static analysis.

The real corpus is **not vendored** (third-party research data with
realistic-looking secret payloads); `fetch_mcptox.sh` pulls it into
`data/` (git-ignored). Without it, Part B runs on the committed synthetic
sample `poison_samples.jsonl` (8 tools faithful to the MCPTox paradigms,
including one benign control and one already-declared-tool-abuse miss).

## Phase 4 — benign-workflow false-positive / specificity (offline)

Detection is half the story: a monitor that also flags legitimate work is
unusable. `run_benign.py` runs benign multi-step workflows on real servers
through the shipped `verify` gate and reports the false-positive rate.

- **hard false-positive** — benign workflow → `verify` exit 1/2. **Must be 0.**
- **soft flag** — a benign action within the declared capability *class* but
  outside the approved policy *scope* → `within_manifest_not_policy`. Not a gate
  failure; reported as policy-tuning noise.

| Workflow | Server | Result |
|---|---|---|
| `fs-workspace` | filesystem (real, 14 tools) | 16-event edit session in `/workspace` → clean |
| `fetch-web` | fetch (real) | fetch an approved host → clean |
| `yfinance-report` | yfinance | two finance-API calls to the approved host → clean |
| `memory-graph` | memory (real) | knowledge-graph ops, no side effects → clean |
| `time-convert` | time (real) | timezone conversion, no side effects → clean |
| `fs-scope-noise` | filesystem (real) | benign read of `/etc/hostname` outside `/workspace` → **1 soft flag, gate still passes** |

Result: **0/6 hard false-positives**, 1/6 soft-flag noise (the deliberate
out-of-scope probe — proof the suite is not rigged trivially green), and the
multi-server **`fleet verify` over 4 benign servers returns exit 0**.

Workflow shapes are modelled on mcpuniverse benign tasks
([arXiv:2512.15163](https://arxiv.org/abs/2512.15163)); four of the five server
manifests are the real public servers in `demo/artifacts/pub_*.json`.

Honest scope note: servers with no external side effects (memory, time) emit no
net/fs/proc events, so the runtime monitor sees only `mcp.call` context markers
— it cannot judge their *internal* correctness, only that they touched nothing
they didn't declare. Driving the full `mcpuniverse` agent loop (LLM + live
servers under the Docker adapter) is the natural extension for real end-to-end
traces.
